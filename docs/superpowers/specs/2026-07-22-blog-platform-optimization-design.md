# 博客平台代码与体验优化设计

日期：2026-07-22
涉及仓库：`myblog-admin`（Vue 前台与管理后台）、`blog_li`（Django REST API）

## 1. 背景与目标

现有系统已覆盖博客浏览、动态发布、分类标签、评论、文件和用户管理，但前台、后台与后端经过多轮增量开发后出现了明显的结构漂移：前端存在重复入口与请求封装、超大页面组件、调试逻辑和视觉规则分散；后端生产配置硬编码、文档与真实目录不一致、部分测试仍引用已经不存在的 `post` 应用。

本轮优化同时改善博客前台和管理后台，目标是：

1. 建立统一、清爽、专业的视觉体系，采用已选定的“现代产品 / 清爽科技”方向。
2. 改善移动端、键盘操作、加载/空/错误状态与内容阅读体验。
3. 收敛前端入口、请求、认证和样式层，降低维护成本。
4. 改善 Django 的环境配置、查询方式、错误响应和有效测试覆盖。
5. 重写开发文档，使安装、配置、测试、构建、部署说明与代码一致。

## 2. 方案选择

评估过三种实施方式：

- 仅做视觉换肤：交付快，但会保留重复请求层、超大组件和后端隐患，长期收益低。
- 渐进式重构并统一设计系统：保留现有技术栈和接口，在核心路径上同步改善体验与结构，风险可控。
- 全量重写：结构最干净，但会带来较高的回归、数据迁移和交付风险。

采用第二种方式。现有 Vue 3、Vite、Pinia、Ant Design Vue、Django 和 DRF 均保留；现有数据库模型和公开 API 路径默认兼容。

## 3. 信息架构与视觉系统

### 3.1 视觉方向

视觉基调为冷白背景、靛蓝主色、深灰正文和低饱和辅助色。界面通过留白、边框和轻阴影区分层级，不使用大面积高饱和渐变或无功能性的动画。

核心设计令牌：

- 主色：`#315BEA`，悬停和浅色背景由同一色阶派生。
- 正文：`#172033`；次要文字：`#667085`；弱提示：`#98A2B3`。
- 页面背景：`#F6F8FB`；卡片背景：`#FFFFFF`；边框：`#E2E7EF`。
- 圆角分为 8、12、16 像素三档；阴影只用于浮层和强调卡片。
- 内容宽度使用受控容器，博客正文最大宽度约 760 像素，列表页最大宽度约 1200 像素。
- 动效时长集中在 160–240ms，并尊重 `prefers-reduced-motion`。

设计令牌落在一个全局主题文件中，并通过 CSS 自定义属性供普通样式、SCSS 和 Ant Design Vue 主题配置共同使用。删除互相覆盖的全局按钮、输入框和标题规则。

### 3.2 博客前台

`BlogLayout` 只负责站点导航、搜索入口、内容插槽、页脚和移动端菜单。复杂的搜索建议拆成独立组件和组合式逻辑，避免布局文件同时承担数据请求和历史记录管理。

博客首页采用：

- 简洁品牌导航和单一搜索入口。
- 首屏介绍区，展示博客定位和主要内容入口。
- 最新内容为主列表，卡片突出标题、摘要、分类、时间和阅读信息。
- 热门文章、分类和标签作为次级信息，不抢占主阅读路径。
- 清晰的骨架屏、空状态、失败重试和分页反馈。

文章详情使用稳定阅读宽度、明确的标题层级、代码块样式、图片约束、上一篇/下一篇和评论区域。分类、标签、搜索和动态页面复用同一套内容卡片与状态组件。

移动端导航使用折叠菜单；搜索改为图标触发的全宽面板；卡片、分页和操作区不产生横向滚动。

### 3.3 管理后台

`DefaultLayout` 形成稳定的应用壳层：可折叠侧栏、顶部栏、面包屑、用户菜单和主内容区。菜单由单一配置生成，不在模板中重复硬编码路由行为。

仪表盘重新组织为：

- 关键指标卡：内容、分类、标签、评论。
- 最近内容和最近评论，提供直接处理入口。
- 趋势图作为次级模块，数据不足时显示解释性空状态。
- 明确的“新建内容”主操作。

列表管理页面统一使用 `PageHeader`、`SearchForm`、`DataTable`、`Pagination` 和确认反馈模式。筛选区可折叠，批量操作只在选中数据后出现，危险操作使用一致的二次确认。

### 3.4 无障碍与响应式

- 所有交互元素提供可见焦点状态、语义标签和不低于 44 像素的移动端点击区域。
- 颜色不作为唯一状态提示，错误、成功和禁用状态同时使用文字或图标。
- 页面主区域、导航和标题保持正确语义层级。
- 关键布局在 360、768、1024 和 1440 像素宽度验证。

## 4. 前端代码架构

### 4.1 单一入口与目录边界

保留 `src/main.js` 作为唯一入口，移除未被构建使用的重复入口和迁移残留。目录职责调整为：

- `api/`：按业务域暴露 API 函数，不保存 UI 状态。
- `services/http/` 或现有 `utils/request.js` 的收敛版本：唯一 Axios 客户端、令牌注入、刷新队列、错误转换和取消逻辑。
- `stores/`：用户和应用级状态，不重复解析多种后端响应格式。
- `components/common/`：无业务或轻业务的可复用展示组件。
- `components/blog/` 与 `components/admin/`：各自领域组件。
- `composables/`：搜索、分页和异步状态等可组合逻辑。
- `styles/`：主题令牌、基础规则、Ant Design 覆盖和页面级样式。

### 4.2 请求与认证

所有 API 通过同一个 Axios 客户端发送。令牌只使用一种存储格式：存储原始 access token，请求拦截器统一添加 `Bearer`。刷新令牌时使用单飞队列，多个并发 401 只触发一次刷新；刷新失败后清理会话并携带当前地址跳转登录页。

客户端只接受后端真实契约，不继续维护多层“猜测响应格式”和把 500 响应改成 200 的兼容逻辑。开发日志由统一的开发开关控制，生产构建不输出令牌、请求体或响应头。

API 错误转换为统一对象：`status`、`code`、`message`、`fieldErrors` 和可选 `requestId`。页面决定显示内联错误、消息提示还是全页失败状态，HTTP 层不直接控制页面加载状态。

### 4.3 页面拆分与状态模型

超过单一职责的布局和页面按“数据获取 / 业务动作 / 展示”拆分。异步页面统一采用四态模型：初始加载、成功、空数据、失败。提交操作单独维护 pending 状态，防止全局遮罩阻塞整个应用。

保留路由级懒加载，但删除 `webpackPrefetch` 注释、重复预加载和固定 500ms 的人为加载延迟。路由守卫只负责认证、标题和滚动行为，不拦截合法的同路径参数变化。

## 5. 后端设计

### 5.1 环境与安全配置

`DEBUG`、`SECRET_KEY`、`ALLOWED_HOSTS`、CORS 来源、数据库、日志级别和媒体/静态路径全部从环境读取并提供安全的开发默认值。生产环境不得默认 `DEBUG=True`、通配 `ALLOWED_HOSTS` 或启用 SQL DEBUG 日志。

移除公开的调试 URL 和仅用于本地排查的测试路由。错误响应保留请求 ID，但不泄露异常堆栈、SQL 或内部配置。

### 5.2 API 与查询

保持现有 `/api/` 和 `/blog/` 路径兼容。列表接口使用明确的分页响应，输入校验失败使用 400，未认证使用 401，无权限使用 403，不存在使用 404，服务异常使用 500。

动态、分类、标签和评论列表根据序列化字段使用 `select_related` / `prefetch_related`，避免逐条查询。计数和点赞更新使用数据库表达式或事务，避免并发下丢失更新。只在真实查询模式需要时增加索引。

### 5.3 测试修复

删除或改写引用不存在的 `apps.post`、旧 URL 和旧响应格式的测试。优先覆盖：

- 登录、刷新、登出和受保护接口。
- 博客动态列表、详情、搜索、分类和标签筛选。
- 点赞、浏览量和评论的权限及并发安全边界。
- 环境配置解析和统一异常格式。

测试默认使用 SQLite 内存数据库，不依赖本机 MySQL。

## 6. 数据流与错误处理

正常数据流：页面或 store 调用领域 API → 单一 HTTP 客户端添加认证和请求 ID → Django 视图校验输入并查询 → 序列化器返回稳定 JSON → 页面进入成功或空状态。

认证过期数据流：接口返回 401 → HTTP 客户端触发一次 refresh → 成功后重放等待请求；失败则清理本地会话并跳转登录页。

异常数据流：Django 异常处理器生成带 `requestId` 的安全错误 → HTTP 客户端转换为统一前端错误 → 表单显示字段错误，列表/详情显示就地失败与重试，非阻断操作使用轻量消息提示。

## 7. 文档重写

两个仓库的 README 与开发文档按真实代码重写，避免复制不存在的目录或命令。

前端文档包含：

- 项目边界（博客前台 + 管理后台）、技术栈和真实目录图。
- Node/npm 版本、环境变量、后端依赖、安装和启动命令。
- 路由、状态、请求层、样式系统和新增组件约定。
- lint、测试、覆盖率、构建和常见故障排查。
- Nginx 部署示例及 `/api/`、`/blog/` 代理说明。

后端文档包含：

- 真实应用目录、Python/MySQL 要求和虚拟环境步骤。
- 完整 `.env.example` 字段说明和开发/生产差异。
- 迁移、创建管理员、测试、静态文件和启动命令。
- API 路径总览、统一响应/错误格式、认证流程。
- Gunicorn/Nginx 部署、安全检查和故障排查。

文档中的命令在干净环境中验证，版本要求以锁文件和实际依赖为准。

## 8. 测试与验收

前端验收：

- `npm test -- --runInBand` 通过，关键请求/认证/组件行为有回归测试。
- `npm run build` 通过，无重复入口和明显的大包警告回归。
- 博客与后台关键页面在桌面和移动宽度无横向溢出。
- 键盘可完成导航、搜索、登录和主要管理操作。
- 加载、空、错误、提交中状态可辨识且不会长期遮挡页面。

后端验收：

- `python manage.py test` 使用测试数据库通过。
- `python manage.py check` 与 `python manage.py check --deploy` 的可操作告警已处理或在文档中说明。
- 列表接口不存在明显 N+1 查询，认证和错误状态码符合约定。
- 生产配置不依赖源码中的密钥、通配主机或强制 DEBUG。

文档验收：所有路径、环境变量和命令与仓库实际内容一致，首次参与开发者可以仅凭 README 和开发指南启动两端。

## 9. 实施顺序

1. 建立回归测试和基线，修正无法运行的测试配置。
2. 收敛主题、全局样式、入口、HTTP 客户端和认证状态。
3. 重构博客壳层、首页、内容卡片与关键状态组件。
4. 重构后台壳层、仪表盘与列表页公共模式。
5. 修正后端环境配置、调试路由、查询和核心 API 测试。
6. 重写两个仓库的 README、开发和部署文档。
7. 执行完整测试、构建、响应式检查和最终 diff 审查。

## 10. 非目标与兼容原则

本轮不更换数据库、不引入 SSR/SSG、不新增 CMS 工作流、不进行数据模型大迁移、不增加与优化目标无关的功能。除非现有接口本身错误且有测试证明，否则不改变公开 URL、字段名称和数据库数据语义。
